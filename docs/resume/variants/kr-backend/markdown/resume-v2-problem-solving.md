## 한 줄 소개

운영 중인 문제를 기술 과제로 다시 정의하고, 지표로 검증해 재발하지 않게 만드는 백엔드 엔지니어입니다.

## 문제 해결 사례

### 사례 1. 실패가 연쇄 전파되는 발송 파이프라인을 분리

- 상황: 업체 1곳 장애가 전체 발송 중단으로 번지면서 월 2-3회 장애가 반복됐습니다.
- 판단: 동기 체인을 유지하면 장애 범위를 줄일 수 없다고 보고 업체 단위 비동기 처리로 전환했습니다.
- 실행: SQS로 큐를 분리하고 PDF 생성은 Python Lambda로 오프로딩했습니다.
- 결과: 발송 실패 월 2-3회 → 0건, PDF 용량 100KB → 10KB, 수기 대응 제거.

### 사례 2. 문서 없는 TMS 연동을 복구 가능한 구조로 전환

- 상황: 재시도와 알림이 없어 타임아웃 발생 시 운영팀이 수동으로 처리했습니다(월 3-5회, 건당 30분).
- 판단: 외부 연동 코드는 테스트 가능성과 장애 가시성이 먼저 확보돼야 한다고 봤습니다.
- 실행: Adapter 분리, Mock 기반 테스트, Retry(최대 3회), Slack 즉시 알림, Audit 로그를 도입했습니다.
- 결과: 수기 대응 월 3-5회 → 0회, 장애 인지 시점 "다음 날" → "즉시".

### 사례 3. 수기 결제 프로세스를 정기결제로 자동화

- 상황: API가 없는 CMS에 월 3회 수기 입력이 필요했고 휴먼 에러가 누적됐습니다.
- 판단: PG 공급사 변경이 반복될 것을 고려해 결제 비즈니스와 PG 의존성을 분리했습니다.
- 실행: 빌링키 정기결제 도입, PG 공통 인터페이스 + Strategy/Adapter 패턴 적용.
- 결과: 월 3회 수기 작업 제거, 신규 PG는 Adapter 1개 추가로 대응.

### 사례 4. 과도한 인프라 비용 구조 개선

- 상황: DocumentDB가 월 140만 원 수준으로 서비스 규모 대비 비정상적으로 높았습니다.
- 판단: 장기 보관 데이터가 스토리지 비용을 밀어 올린다고 판단하고 수명주기 정책을 도입했습니다.
- 실행: TTL Index로 1개월 초과 데이터 자동 만료, 만료 전 S3 백업.
- 결과: 월 140만 원 → 12만 원(91% 절감), 데이터 유실 0건.

### 사례 5. 다공급사 데이터 불일치 문제 해결

- 상황: 동일 지점이 공급사별 다른 이름으로 저장돼 검색/비교 정확도가 낮았습니다.
- 판단: 실시간 교정보다 마스터 데이터 정규화가 장기적으로 유지비가 낮다고 판단했습니다.
- 실행: 마스터 지점 + 공급사 매핑 테이블 설계, 일괄 번역 파이프라인 운영.
- 결과: 타깃 지역 일 평균 예약 200건 → 400건.
